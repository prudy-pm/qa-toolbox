The Governing Rule
Default to the lowest layer that can honestly prove the behaviour.
⚙️
Unit
Pure logic, in-process, no I/O. Dev-owned. QA asks about it when needed, but does not write it.
🔗
Integration
API contracts, DB writes, events, persistence, state transitions. This is a QA workhorse.
🌐
E2E / Automation
User-visible behaviour, journeys, rendering, auth/session behaviour. Keep this layer thin and deliberate.
Rule: If the browser is not the only honest proof, it probably does not need to be an E2E test.
If everything becomes E2E, the browser becomes your therapist and your bottleneck.

What QA Drives, Influences, and Needs
Clear boundaries stop work from being absorbed badly or not done at all. This deck describes capability and approach — not automatic ownership of every test type.

QA Drives

  • Functional automation strategy — recognising when E2E, Integration, or smoke coverage is needed and what good looks like
  • Test case design — happy path, edge cases, negative paths
  • Coverage tracking — Covered / Partial / Pending / Out of Scope
  • Bug logging and triage — including AI-assisted workflows
  • Suite stability — anti-flake standards, speed tiering, maintenance

QA Influences

  • Requirement quality — can this be tested from what is written?
  • PR review conversations — is the right layer covered?
  • Definition of Done — what quality means before merge or release
  • Testability hooks — data-testid, admin seams, stable selectors
  • Unit test coverage conversations — Dev owns, QA can query

Dev Owns / QA Needs

  • Unit tests — pure logic belongs with Dev
  • Code quality — linting and implementation standards
  • Testability hooks — reliable locators and helper seams in the app
  • Bug fixes — linked back to failing TCs or review findings
  • Shared dependency support where the project requires emulators or stubs that QA must be able to trust

The first job is orientation, not automation.

Find where requirements live — wiki, tickets, linked docs, or scattered tribal knowledge

6️⃣

Trust Boundaries — Real / Emulated / Stubbed
This is one of the fastest ways to stop testing the wrong thing in the wrong environment.
ComponentTypeMeaning
Frontend / UIRealActual application code or services you own and can run as they really behave
API / ServicesRealActual application code or services you own and can run as they really behave
Database / Queue / StorageEmulatedInfrastructure or dependencies you rely on that can be run locally with sufficient fidelity
Email / SMS / Payment / External providerStubbedDependencies you cannot call deterministically or faithfully enough in the main suite
Workshop prompt: Fill in this table for your current project. If you cannot classify a component, that uncertainty is already telling you something useful.

Shared Test Dependency Readiness
If a project relies on emulated or stubbed dependencies, QA needs more than access — QA needs reliability.
ExpectationWhy it matters
Starts reliablyA strategy cannot depend on something that works inconsistently
Can be reset predictablyRepeatable testing needs a known state
Behaviour is documentedQA needs to know what is real, approximated, or unsupported
Failure modes are visibleEnvironment problems should not masquerade as product bugs
Data can be seeded / controlledRepeatable setup depends on this
Limits are knownUnsupported behaviour should not be mistaken for coverage
Practical framing: another party may provision the emulator or stub, but QA should not quietly rely on it unless readiness expectations are explicit.
Rule: If a dependency is not test-ready, it should not sit inside the strategy as though it is.

When to Use Integration vs E2E
The most practical layer decision QA makes repeatedly. If this coverage needs to exist, this is the best way to approach it.
🔗
Use Integration when you need to prove…
  • API contracts
  • DB writes / persistence
  • message or event flows
  • state transitions
  • ETL / transformation
  • eventual consistency
🌐
Use E2E when you need to prove…
  • the user can see it
  • the user can do it
  • the journey works across pages
  • rendering / layout matters
  • error / empty states degrade correctly
  • browser session behaviour matters
Rule: If API state or stored data is enough proof, prove it lower and let the browser prove only what the browser genuinely owns. Ownership of implementation may still vary by team.

4-State Coverage Taxonomy
“Not tested” is too vague to be useful.
StateMeaningAction
CoveredAutomated and passingMaintain
PartialSome covered, gaps remainFinish or explicitly accept
PendingCan be automated here, not yet builtEstimate and schedule
Out of ScopeCannot run in this environmentDocument and revisit only if constraints change
Most important distinction: Pending is backlog. Out of Scope is an environment or dependency constraint. They are not the same thing.

Definition of Ready
The intake gate for both QA and AI-assisted workflows. Ticket-side readiness and QA enrichment are not the same thing.
FieldPriorityWhy it matters
Feature titleMust haveDefines the scope boundary
Where — page / screen / route / platformMust haveAI and QA both need location
Relevant linked contextMust haveGives QA and AI enough structured context to reason correctly without tribal knowledge
Acceptance criteriaMust haveBecomes test cases
Out of scopeMust havePrevents over-generation and scope creep
Existing data behaviourShould haveProtects backward compatibility
Negative pathsShould haveImproves edge-case quality
Design referenceShould haveSupports UI validation
Test data requirementsShould haveImproves repeatability and setup quality
Ticket-side DOR: authors provide the minimum structured context needed to understand the change.
QA intake enrichment: QA derives likely systems affected, likely test layers, likely automation route, and likely data needs.
Why this matters for Claude Code: vague requirements produce generic output, generic TCs, and weak PR review. Better input quality is the cheapest way to improve AI usefulness.

Weak vs Ready Requirement Example
A developer with context can infer. Claude Code cannot. A QA engineer joining later cannot either. Acceptance criteria anchor test design — they do not replace QA's smart testing.
Weak

“Remove the referral code field.”

Not enough to update automation, generate useful TCs, or review a PR against expected outcomes.
Ready

Specifies where the change lives, what layers it touches, what the observable outcomes are, how existing data behaves, and at least one meaningful negative path.

DOR Ticket Comment Template
A reusable intake response when a ticket is not ready to test from.
Definition of Ready — please complete before sprint pull

• Where: page/screen + URL path + platform(s)
• Scope: UI only? API contract? DB? Analytics? CRM mapping?
• Acceptance criteria (Given/When/Then)
• Existing data: behaviour for records/accounts already holding the field/value
• Adjacent UI: layout, validation, tab order, button enable state
• Negative paths: deep links, query params, role variants, partial sessions
• Design reference: Figma or before/after screenshot
• Out of scope: explicit list
• Test data requirements: role / account state / seed data
Workshop prompt: Take a real ticket and run this comment against it. If it feels too long, the ticket is usually under-specified.

QA PR Change Assessment
Use AI, MCP, and PR artefacts to understand what changed before waiting for full deployment-level testing.
What it does
✅
Reads PR artefacts through MCP
✅
Compares changes against linked requirement context
✅
Checks consistency against QA test understanding
✅
Flags likely functional mismatches and automation impact
✅
Gives QA an earlier signal before full execution
What it does not do
❌
Replace developer code review
❌
Judge code style or syntax
❌
Act as a developer merge gate
❌
Approve or reject the PR
❌
Replace proper QA execution
The goal is not to review code like a developer. The goal is to review change like QA.

Day-One Bootstrap Checklist
Scaffold first. Prove one slice. Scale second.
1️⃣
Decide real / emulated / stubbed for every major component
2️⃣
Create scaffold folders — page objects, support layer, or fixtures / builders
3️⃣
Add base readiness policy — no ad hoc waits
4️⃣
Add route constants — zero raw URLs in tests
5️⃣
Add one builder for the primary entity under test
6️⃣
Add isolation and cleanup helper or fixture
7️⃣
Configure realistic timeouts and artefacts
8️⃣
Add speed tags — fast / slow or equivalent
9️⃣
Get one thin slice green before scaling
Heuristic: if a new test takes 30 lines of setup, the scaffold needs work — not the engineer.

Stability & Anti-Flake Checklist
A stable suite is a usable suite.
⏱️
Timing & readiness
  • no sleeps
  • no network-idle
  • semantic readiness only
  • realistic timeouts
🔒
Isolation
  • unique IDs
  • no shared mutable state
  • cleanup in finally
  • parallel only when honest
🎯
Assertions & locators
  • prove persisted outcomes
  • not just transient UI
  • scoped role/name or data-testid
  • no brittle selectors
🧪
Environment honesty
  • do not mislabel smoke as deterministic coverage
  • warm the environment before judging failures
  • real third-party seams belong in thin smoke lanes

Flake Triage Sequence
When a test fails intermittently, check in this order.
1️⃣
Timing — readiness, cold starts, asynchronous settle time
2️⃣
Isolation — shared state, reused data, cross-test contamination
3️⃣
Assertion type — proving something too transient or indirect
4️⃣
Locator ambiguity — wrong region, reused label, unstable selector
5️⃣
Environment — broken emulator, unhealthy dependency, config drift
Rule: quarantine confirmed flakes, fix root cause, and do not extend sleep and call it stability.

Test Layers vs Quality Domains
Do not confuse how something is proven with what kind of risk is being evaluated.
Test LayersMeaning
UnitHow pure logic is proven
IntegrationHow seams, contracts, persistence, and state are proven
E2EHow visible user behaviour is proven
SmokeHow narrow reachability / sanity checks are proven
Quality DomainsMeaning
FunctionalCorrectness of behaviour
PerformanceResponsiveness and efficiency
Load / ScalabilityBehaviour under traffic or demand
AccessibilityUsability for broader user needs
Reliability / ResilienceRecovery, robustness, and fault behaviour
SecurityProtection against misuse or exposure
Key distinction: test layers describe how behaviour is proven. Quality domains describe what kind of risk is being evaluated.

Final QA Working Model
The goal is not to automate everything. The goal is to automate the right things, at the right layer, early enough to matter.
📍
If assigned to a project
  • run trust boundary classification
  • apply DOR
  • create coverage map
  • choose first useful quality gate
  • scaffold before scaling
🧰
If not yet assigned
  • keep templates ready
  • prepare scaffold defaults
  • know your tool stack
  • know your entry-point questions
  • adapt, do not blindly copy
Maturity rule: each escaped defect should trigger a decision — should this now be protected by automation, and at which layer?
Outcome: a QA engineer should be able to pick up this deck, look at a project, and know how to start removing manual effort in a structured way.
